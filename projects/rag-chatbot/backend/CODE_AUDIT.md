## Directory: `src/`

### `src/index.ts`

**Summary:**
Application entry point. Bootstraps the Express server, registers global middleware (CORS and JSON body parsing up to 10 MB), mounts the two API routers, and exposes the `app` as a default export for test consumption. The `app.listen()` call is guarded by `NODE_ENV !== 'test'` to prevent port binding during unit tests.

**Exported symbols:**
| Symbol | Kind | Purpose | Parameters | Returns |
|--------|------|---------|------------|---------|
| `default` (app) | `express.Application` | The configured Express app used by tests and the server | — | `express.Application` |

**External dependencies:**
| Package | Reason |
|---------|--------|
| `dotenv` | Loads `.env` into `process.env` before any other import |
| `express` | HTTP server framework |
| `cors` | Global CORS middleware — allows all origins by default |
| `./routes/ingest` | Mounts ingest route at `/api/ingest` |
| `./routes/chat` | Mounts chat route at `/api/chat` |

**Connection to other files:**
- Imported by all test files that need a full app instance (`__tests__/index.test.ts`, `__tests__/e2e/full-flow.e2e.test.ts`, etc.)
- Delegates all business logic to `routes/chat.ts` and `routes/ingest.ts`

**Potential improvements / code smells:**
1. **CORS is fully open** — `cors()` with no options allows any origin. For a production deployment, this should be restricted to the known frontend origin.
2. **No global error handler** — Express 4 requires an `(err, req, res, next)` middleware to catch unhandled errors. Without it, unhandled promise rejections in async route handlers that Express 5 would catch are silently swallowed in Express 4 (see also the bug noted in `security.test.ts` for non-string `filePath`).
3. **No request logging** — There is no access log (e.g., `morgan`), making production debugging difficult.
4. **PORT may be `NaN`** — `parseInt('', 10)` returns `NaN` which `app.listen()` silently ignores; a non-numeric `PORT` env var will cause the server to bind on a random port. Should validate with `Number.isNaN`.

**Potential improvements / code smells:**

3. **No pool configuration** — No `max`, `idleTimeoutMillis`, or `connectionTimeoutMillis` settings are specified; defaults may be inappropriate for production load.

### `src/db/schema.sql`

**Summary:**
DDL that creates the `document_chunks` table and its HNSW index. Requires the `vector` PostgreSQL extension (pgvector). The embedding column is typed as `vector(3072)` for Vertex AI gemini-embedding-001 output. The HNSW index uses a `halfvec(3072)` cast because pgvector's HNSW implementation caps at 2000 dimensions for `vector` but supports up to higher dimensions via `halfvec`. Cosine similarity ops are used.

**Schema:**
```sql
document_chunks (
  id           SERIAL PRIMARY KEY,
  content      TEXT NOT NULL,
  embedding    vector(3072),
  source_file  TEXT,
  chunk_index  INTEGER,
  created_at   TIMESTAMPTZ DEFAULT NOW()
)
```

**Connections:** Applied once at database setup; referenced by `db/client.ts`, `routes/ingest.ts`, and `services/retrieval.ts`.

**Potential improvements / code smells:**
1. **`embedding` is nullable** — The column has no `NOT NULL` constraint, so a chunk can be stored without an embedding, making it invisible to similarity search.
2. **No unique constraint on `(source_file, chunk_index)`** — Re-ingesting the same file will duplicate all chunks rather than upsert. This can cause unbounded table growth.
3. **No index on `source_file`** — Filtering or deleting by source file (used in tests) requires a full table scan.

### `src/db/migrate_to_3072.sql`

**Summary:**
One-time migration script for upgrading from a previous embedding provider (Voyage AI, 512-dim) to Vertex AI gemini-embedding-001 (3072-dim). Since pgvector does not support in-place vector dimension changes, this script deletes all existing chunks, drops the old column and index, and recreates both at 3072 dimensions.

**Connections:** Applied manually by an operator; not referenced by application code.

**Potential improvements / code smells:**
1. **Destructive — no backup step** — Running this script deletes all data. A comment recommending a `pg_dump` before running would reduce risk.
2. **Not idempotent** — Running the script twice will fail on the `DROP INDEX IF EXISTS` (already removed) and `ALTER TABLE DROP COLUMN` (already removed). Consider wrapping in a transaction with guards.

### `src/routes/chat.ts`

**Summary:**
Express router handling `POST /api/chat`. Implements the RAG query pipeline: validates the incoming `query` string → generates a query embedding → retrieves the top-5 most similar document chunks → streams the LLM response as Server-Sent Events (SSE). When no chunks are found (no documents ingested), it returns a friendly SSE message without calling the LLM. All errors before SSE headers are sent produce a JSON `500`; errors after headers are sent produce an SSE `error` event.

**Connection to other files:**

3. **SSE connection cleanup** — There is no `req.on('close', ...)` handler to detect if the client disconnects mid-stream, which would leave the LLM streaming response running unnecessarily.

4. **No deduplication** — Re-ingesting the same file appends duplicate chunks to the database.
5. **Individual INSERT per chunk** — Using `COPY` or a multi-row `INSERT ... VALUES` would be significantly faster for large documents.

### `src/services/pdf.ts`

**Summary:**
Handles PDF text extraction and text chunking. Uses `pdf-parse` to extract raw text from a PDF buffer. Then chunks the normalized text into overlapping segments of approximately 500 tokens (2000 characters), preferring paragraph boundaries, then sentence boundaries, then hard character cutoffs. Exports three functions: a low-level PDF reader, a pure chunking function, and a combined helper.

**Exported symbols:**
| Symbol | Kind | Purpose | Parameters | Returns |
|--------|------|---------|------------|---------|
| `Chunk` | Interface | Shape of a text chunk | `content: string`, `chunkIndex: number` | — |
| `parsePdf` | `async function` | Reads a PDF file and returns the full extracted text | `filePath: string` | `Promise<string>` |
| `chunkText` | `function` | Splits text into overlapping chunks | `text: string` | `Chunk[]` |
| `parsePdfIntoChunks` | `async function` | Combines `parsePdf` and `chunkText` | `filePath: string` | `Promise<Chunk[]>` |

**Internal constants:**
| Constant | Value | Purpose |
|----------|-------|---------|
| `CHUNK_TOKENS` | 500 | Target tokens per chunk |
| `OVERLAP_TOKENS` | 50 | Token overlap between consecutive chunks |
| `CHARS_PER_TOKEN` | 4 | Characters-per-token approximation for English text |
| `CHUNK_CHARS` | 2000 | Derived max chars per chunk |
| `OVERLAP_CHARS` | 200 | Derived overlap in characters |

**External dependencies:**
| Package | Reason |
|---------|--------|
| `fs` | `fs.readFileSync()` to load PDF bytes into a `Buffer` |
| `pdf-parse` | Extracts text content from a PDF buffer |

**Connection to other files:**
- Called by `routes/ingest.ts` (`parsePdfIntoChunks`)
- `Chunk` type is used in `routes/ingest.ts`
- Tested in `__tests__/pdf.test.ts`

**Potential improvements / code smells:**
1. **Synchronous `readFileSync`** — For large PDFs this blocks the event loop. Using `fs.readFile` (callback) or `fs.promises.readFile` would be non-blocking.
2. **Character-based token approximation** — The 4 chars/token ratio is a rough estimate and breaks down for non-English text, code, or heavily punctuated content. A real tokenizer (e.g., `tiktoken`) would give accurate chunk boundaries.
3. **`chunkText` overlap start calculation** — `start = Math.max(start + 1, end - OVERLAP_CHARS)` can create an infinite loop if `OVERLAP_CHARS >= CHUNK_CHARS` (i.e., if `end - OVERLAP_CHARS <= start + 1`). With current constants (OVERLAP=200, CHUNK=2000) this is safe, but changing constants without updating the guard could cause an infinite loop.
4. **No minimum chunk size filter** — Very short trailing chunks (e.g., a single newline or period) can be emitted and stored as noise.
5. **No multi-column or scanned PDF handling** — `pdf-parse` struggles with complex layouts (columns, tables, scanned images). The caller receives garbled text without any indication.

### `src/services/retrieval.ts`

**Summary:**
Queries the `document_chunks` table in PostgreSQL using pgvector's cosine distance operator (`<=>`) to find the most similar chunks to a query embedding. Uses a dedicated pool client within a transaction to set `ivfflat.probes = 10` before the similarity search (ensuring all IVFFlat clusters are checked on small datasets). Returns results mapped to the `ChunkResult` interface with `similarity` parsed from PostgreSQL's string representation to a JavaScript `float`.

**Exported symbols:**
| Symbol | Kind | Purpose | Parameters | Returns |
|--------|------|---------|------------|---------|
| `ChunkResult` | Interface | Shape of a retrieved chunk | `id: number`, `content: string`, `sourceFile: string \| null`, `chunkIndex: number \| null`, `similarity: number` | — |
| `retrieveTopChunks` | `async function` | Retrieves top-K similar chunks by cosine similarity | `embedding: number[]`, `topK: number = 5` | `Promise<ChunkResult[]>` |

**External dependencies:**
| Package/Module | Reason |
|----------------|--------|
| `../db/client` | Acquires a dedicated `PoolClient` for transactional query |

**Connection to other files:**
- Called by `routes/chat.ts`
- `ChunkResult` type imported by `services/llm.ts`
- Tested in `__tests__/retrieval.test.ts`

**Potential improvements / code smells:**
1. **`SET LOCAL ivfflat.probes`** — The schema now uses HNSW (not IVFFlat), so this session-level setting is a no-op. The comment and code are stale and should be cleaned up. HNSW does not require probe tuning.
2. **Transaction for a read-only query** — `BEGIN ... COMMIT` is not necessary for a `SELECT`; a simple `SET` + `SELECT` on a dedicated client (without transaction) achieves the same result with less overhead. Alternatively, just call `pool.query()` directly since HNSW needs no probe setting.
3. **Vector literal via string interpolation** — `[${embedding.join(',')}]` could theoretically be very long for 3072-dim vectors; using a parameterized array instead of a string literal would be more idiomatic, though the current approach works with the `::vector` cast.
4. **No similarity threshold filtering** — All top-K results are returned regardless of their similarity score, which may include very low-relevance chunks when the document corpus is small or unrelated to the query. A minimum similarity threshold (e.g., `WHERE 1 - (embedding <=> ...) > 0.5`) would improve answer quality.
5. **Hard-coded `topK` default** — The default of `5` is appropriate but should ideally be configurable.

## Cross-cutting Observations

### Architecture Summary
The application is a clean three-layer RAG pipeline:
1. **Ingest path**: `POST /api/ingest` → PDF parse → chunk → embed (Vertex AI) → store (pgvector)
2. **Query path**: `POST /api/chat` → embed query (Vertex AI) → retrieve top-K (pgvector cosine) → generate answer (Gemini streaming) → SSE to client

### Security Posture
- **SQL injection**: Queries use parameterized statements (`$1`, `$2`); the only raw string interpolation is the vector literal `[${embedding.join(',')}]`, which is safe since embedding values are numbers.
- **Path traversal**: `path.resolve()` normalizes the path but does not restrict it to a safe directory. Any valid server-side path can be read.
- **XSS**: The backend is a JSON/SSE API; HTML injection is a frontend concern.
- **Prompt injection**: Not mitigated server-side; relies on the LLM's system prompt grounding.

### Test Coverage
- Unit tests cover all services and routes with mocked dependencies.
- Integration tests cover DB layer and API-DB interaction (skipped without a real DB).
- E2E tests cover the full request pipeline end-to-end (with mocked external APIs).
- Security tests document known vulnerabilities (path traversal, async error bug).

### Critical Bugs
1. **Non-string `filePath` crash** (`routes/ingest.ts:22`) — `typeof filePath !== 'string'` check runs before `!filePath`, so truthy non-strings (e.g., `{}`  or `[]`) pass the guard and crash `path.resolve()` outside the try/catch. Express 4 will not catch this.
2. **Stale `ivfflat.probes` setting** (`services/retrieval.ts:37`) — The schema uses HNSW, not IVFFlat. This setting is harmless but misleading dead code.

### Missing Features / Gaps
- No authentication or API key protection on any endpoint.
- No rate limiting.
- No document management API (cannot list, delete, or update ingested documents).
- No streaming backpressure handling (client disconnect not detected).
- `service_key.json` is loaded synchronously at import time, requiring filesystem mocking in every test file — a significant testing friction point.
